# 🐷 AI 개발 지침 (AI Guidelines)

이 문서는 **걸음수 기반 청소년 활동량 증진 앱** 프로젝트를 개발하는 AI 개발자(LLM)를 위한 지침서입니다. 코드 작성, 파일 생성, 라이브러리 추가 등 모든 개발 프로세스에서 이 지침을 최우선으로 준수해야 합니다.

---

## 1. 📌 프로젝트 개요 및 아키텍처

- **목적**: 청소년들이 자발적으로 신체활동량을 늘릴 수 있도록 돕는 게임화(Gamification) 모바일 앱
- **핵심 동기부여 요소**:
  1. **사회적 동기부여**: 목표 걸음수 미달 시 연락처로 경고 메시지 발송
  2. **시각적 동기부여**: 누적 걸음수에 따른 단계별 캐릭터 진화 (돼지 → 예쁜/잘생긴 캐릭터)
  3. **스토리텔링 동기부여**: 걸음수로 해금되는 AI 인터랙티브 스토리 (선택지 포함)
- **대상 플랫폼**: React Native (Expo) - iOS 및 Android 호환

---

## 2. 🛠 기술 스택 및 개발 원칙

AI는 다음 라이브러리와 서비스 구성을 기준으로 코드를 작성해야 합니다.

| 기술 영역 | 사용 도구 / 라이브러리 | 구현 가이드라인 |
|---|---|---|
| **프레임워크** | Expo (React Native) | `App.js` 및 `screens/` 폴더 기반의 화면 구조 준수 |
| **걸음수 측정** | `expo-sensors` (Pedometer) | 실시간 및 일일 누적 걸음수 가져오기 구현 |
| **연락처 권한** | `expo-contacts` | 연락처 권한 요청 및 사용자 선택 연락처 목록 저장 |
| **메시지 발송** | `expo-sms` | `SMS.sendSMSAsync`를 활용한 시스템 문자 전송 창 열기 |
| **백엔드/DB** | Firebase (Auth + Firestore) | 사용자 정보, 일일 걸음수, 획득 포인트, 스토리 진행 상황 기록 |
| **AI 스토리** | Google AI Studio (Gemini SDK) 또는 Claude API | 연속성 있는 스토리 생성을 위한 히스토리 프롬프트 설계 |

---

## 3. ⚠️ 핵심 제약 사항 및 구현 원칙 (필독)

AI가 코드를 설계할 때 반드시 지켜야 할 기술적 제약 사항입니다. 이 원칙을 어기는 설계는 허용되지 않습니다.

### 3.1. SMS 자동 발송 제약 (스팸 방지 및 OS 정책)
- **절대 원칙**: 앱에서 백그라운드로 사용자의 개입 없이 문자(SMS)를 **자동 발송하는 것은 불가능**합니다. (iOS/Android 보안 정책)
- **구현 방식**: 목표 걸음수 미달이 감지되었을 때, `expo-sms`를 통해 **OS 기본 문자 메시지 작성 화면**을 띄우고, 수신인과 내용이 자동으로 입력된 상태에서 **사용자가 직접 '전송' 버튼을 누르도록 설계**합니다.
- **기획적 승화**: 이 수동 발송 구조를 '실수 방지 및 개인정보 보호를 위한 의도된 설계'로 발표 자료와 코드 주석에 설명해 둡니다.

### 3.2. Pedometer(만보기) 백그라운드 제약
- **절대 원칙**: Expo의 기본 `Pedometer` API는 앱이 완전히 종료되었거나 백그라운드 상태일 때 실시간 걸음수 이벤트를 수신하지 못할 수 있습니다.
- **우회 전략**: 앱이 백그라운드에서 실시간 감시하게 만들기보다는, **"사용자가 저녁 시간대에 앱을 켰을 때"** 혹은 **"사용자가 앱을 열어 활성화할 때"** 당일 총 걸음수를 체크하고, 미달성 시 알림(Push or Local Notification) 및 문자 작성 UI를 유도하는 배치성 이벤트 흐름으로 설계합니다.

### 3.3. 캐릭터 진화 시스템
- **절대 원칙**: 캐릭터 변화는 서버 비용 최소화를 위해 **로컬 이미지 에셋**을 사용합니다.
- **구현 방식**: `assets/characters/` 폴더에 캐릭터 단계별 이미지(`stage0.png`, `stage1.png` 등)를 정의하고, Firestore의 누적 걸음수 필드값을 조건부 렌더링하여 표시합니다.

### 3.4. AI 스토리 생성 및 프롬프트 관리
- **절대 원칙**: 이전 스토리의 연속성을 보장해야 합니다.
- **구현 방식**:
  - API 호출 시 단순히 "다음 이야기를 지어줘"가 아니라, **이전 이야기 요약 및 사용자 선택지 이력(Firestore 저장)**을 시스템 프롬프트(Context)에 포함하여 전달해야 합니다.
  - API 키는 클라이언트 코드에 직접 하드코딩하지 않고, `EXPO_PUBLIC_` 환경변수 또는 안전한 방식으로 주입받도록 안내해야 합니다.

## 4. 📂 폴더 구조 및 파일 역할 정의

코드를 제안하거나 생성할 때는 다음 구조를 엄격히 준수하여 해당 역할에 맞는 파일에 로직을 작성합니다.

```
MyApp/
├── App.js                   # 네비게이션 및 전체 레이아웃 구성
├── screens/                 # 사용자 화면
│   ├── HomeScreen.js        # 오늘 걸음수, 캐릭터 일러스트, 진행 상황 표시
│   ├── StoryScreen.js       # AI 스토리 뷰어, 포인트 선택지 잠금 해제 화면
│   └── SettingsScreen.js    # 목표 걸음수 설정, 경고 문자용 연락처 목록 관리
├── services/                # 유틸리티 및 외부 API 연동 모듈
│   ├── pedometer.js         # 걸음수 측정 및 권한 획득 모듈
│   ├── firebase.js          # Firestore DB 읽기/쓰기 및 회원가입/로그인 처리
│   ├── aiStory.js           # Gemini/Claude API 호출 및 프롬프트 구성
│   └── smsAlert.js          # 연락처 무작위 추출 및 SMS 전송 UI 호출 로직
└── assets/
    └── characters/          # 돼지부터 단계별 캐릭터 성장 이미지
```

## 5. 💡 코드 스타일 및 개발 에티켓

1. **상태 관리**: 단순 앱이므로 React의 `useState`, `useEffect`와 Context API를 활용하여 전역 상태(걸음수, 포인트, 캐릭터 상태 등)를 가볍게 관리합니다.
2. **주석 작성**: 코드 작성 시 핵심 분기(예: SMS 발송 팝업 시점, Pedometer 백그라운드 제약 주석 등)에는 한국어로 상세한 주석을 작성합니다.
3. **비동기 처리**: `async/await` 구조와 적절한 `try-catch` 예외 처리를 적용하여 네트워크 오류나 권한 거부 상황에서도 앱이 강제 종료되지 않도록 방지합니다.
4. **한글 출력**: 사용자 피드백 메시지, 오류 안내, UI 텍스트는 모두 친근한 **한국어**로 작성합니다.
